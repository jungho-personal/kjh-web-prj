Personal & Development Blog and Portfolio (kjh-web-prj)

==================================================

0. Purpose of This Document

이 문서는 kjh-web-prj 프로젝트의 “지침(Guideline)”이다.

이 문서의 목적은 다음과 같다.

- 프로젝트의 목표와 방향을 명확히 한다.
- 기술 및 구조 선택의 판단 기준을 고정한다.
- 시간이 지나도 드래프트가 생기지 않도록 한다.
- 프로젝트 관련 모든 분석과 의사결정의 기준점을 제공한다.

이 문서는 다음을 다루지 않는다.
- 현재 코드 구조
- 파일/디렉토리 구성
- 구현 상태
- 런타임 설정 및 환경변수
- 배포 상세

위 내용은 PROJECT_CONTEXT.txt 문서에서 단일 기준으로 관리한다.

==================================================

1. Project Purpose

이 프로젝트는 개인 프로젝트로서 다음 목적을 가진다.
- 개인 개발 이력과 사고 과정을 기록한다.
- Blog / Dev Log / Portfolio를 하나의 웹 서비스로 통합한다.
- 실제 클라우드 환경에서 서비스 배포 및 운영 경험을 축적한다.
- 과도한 엔터프라이즈 설계 없이 현실적인 개인 서비스 구조를 실험한다.

==================================================

2. Project Scope

[In Scope]
- Blog 및 Dev Log 콘텐츠 관리
- Portfolio 정적 콘텐츠 제공
- 개인 관리자(Admin)를 통한 콘텐츠 작성 및 수정
- 클라우드 환경에서의 실제 운영

[Out of Scope]
- 다중 사용자 시스템
- 외부 사용자 인증 및 권한 관리
- 고가용성 및 대규모 트래픽 대응
- 엔터프라이즈 수준의 보안/컴플라이언스
- 개인 프로젝트 규모를 벗어나는 요구는 설계 대상이 아니다.

==================================================

3. Architecture Principles

3.1 Single Server Principle

- Frontend와 Backend는 하나의 서버에서 통합 운영한다.
- 서버를 분리하여 발생하는 복잡도는 이 프로젝트의 목표에 부합하지 않는다.

3.2 Simplicity First

- 가능한 한 단순한 구조를 유지한다.
- “나중에 필요할지도 모른다”는 이유로 복잡도를 추가하지 않는다.
- 리스트성 데이터 조회는 초기 로딩 성능을 우선하며,
  필요한 범위만 점진적으로 로딩하는 구조(cursor / load-more 등)를 기본 원칙으로 한다.

3.3 Code Over Tool

- 특정 도구나 플랫폼에 종속된 구조를 지양한다.
- 도구는 언제든 교체 가능해야 한다.

==================================================

4. Technology Selection Policy

기술 선택 시 다음 기준을 따른다.
- 개인이 장기간 유지보수 가능한가
- 실제 운영 환경에서 검증 가능한가
- 설정 및 러닝커브가 과도하지 않은가
- 문제 발생 시 원인 파악이 가능한가

위 기준에 부합하지 않는 기술은 도입하지 않는다.

==================================================

5. Hosting & Infrastructure Policy

5.1 Hosting Platform

- Railway를 기본 운영 플랫폼으로 사용한다.

선택 이유:
- 배포 및 운영의 단순성
- 서버 슬립 없음
- 데이터베이스 유지 관리 편의성

5.2 Cloud Provider Policy

- 개인 프로젝트에서 직접 보안, 네트워크, SSL 등을 구성하는 것은 과도하다고 판단한다.
- AWS, GCP와 같은 IaaS는 학습 목적이 아닌 이상 선택하지 않는다.

==================================================

6. Admin & Security Policy

- Admin 기능은 개인 전용이다.
- 외부 사용자 또는 다중 관리자 시나리오는 고려하지 않는다.
- 보안은 “충분한 수준”을 목표로 하며, 완벽함을 추구하지 않는다.
관리 비용이 급증하는 설계는 지양한다.

==================================================

7. Documentation Policy

7.1 Document Roles

각 문서는 명확한 역할을 가진다.

- Project Guideline:
  프로젝트의 목적, 원칙, 판단 기준을 정의한다.
  
- PROJECT_CONTEXT.txt:
  현재 코드 구조, 런타임 규칙, 운영 상태에 대한 사실(Facts)을 기록한다.
  
- Dev Log:
  모든 작업의 기록과 변경 이력을 남긴다.

문서의 역할을 혼합하지 않는다.

7.2 Mandatory Reference Rule (중요)
PROJECT_CONTEXT.txt의 공식 최신 버전은 GitHub 저장소에 커밋된 파일을 기준으로 한다.

GitHub URL:
https://github.com/jungho-personal/kjh-web-prj/PROJECT_CONTEXT.txt

판단 기준은 항상 GitHub에 커밋된 최신 버전을 따른다.

프로젝트와 관련된 모든 작업은 다음 규칙을 반드시 따른다.

- 분석, 설계, 답변, 코드 작성, 리팩토링, 디버깅을 수행할 때
  항상 PROJECT_CONTEXT.txt를 최우선 기준으로 참고한다.

- PROJECT_CONTEXT.txt와 충돌하는 가정이나 설명은 허용하지 않는다.

- PROJECT_CONTEXT.txt에 명시되지 않은 내용에 대해 판단이 필요한 경우:
  1. PROJECT_CONTEXT.txt를 먼저 업데이트한다.
  2. 그 이후 작업을 진행한다.

이 규칙은 예외 없이 적용한다.

7.3 Update Rules

- Project Guideline:
  프로젝트의 목적, 원칙, 판단 기준이 변경될 때만 수정한다.

- PROJECT_CONTEXT.txt:
  코드 구조, 런타임 규칙, 운영 사실이 변경되면 즉시 수정한다.

- Dev Log:
  모든 작업에 대해 반드시 작성한다.

==================================================

8. Change Management Rule

다음 변경이 발생하면 반드시 기록한다.
- 구조적 의사결정 변경
- 운영 방식 변경
- 기존 원칙을 깨는 예외 도입

기록 순서는 다음과 같다.
1. Dev Log
2. PROJECT_CONTEXT.txt (필요 시)
3. Project Guideline (원칙 변경 시)

==================================================

9. Guideline Update Rule

이 문서는 다음 경우에만 수정한다.

- 프로젝트 목적이 변경될 때
- 아키텍처 원칙이 변경될 때
- 기술 선택 기준이 변경될 때

단순 구현 변경으로는 수정하지 않는다.

==================================================

10. Final Principle

이 프로젝트의 모든 설계 결정은 다음 질문을 기준으로 한다.

“지금의 나 혼자, 장기간 유지할 수 있는가?”

이 질문에 YES가 아니라면, 해당 선택은 하지 않는다.

==================================================

사용자가 "작업 마무리하자"는 말 혹은, "작업 로그 남기자" 와 같은 작업 로그 남기는 내용과
관련된 말을 하면, 작업한 내용을 아래 템플릿에 맞춰 바로 복붙가능하게 txt 로 작성해야 한다.
